# 缺失功能优先级与联调开发设计文档

**日期：** 2026-04-21
**范围：** mybill-app 缺失功能梳理、优先级划分、前后端联调开发
**方案：** 分阶段联调文档

---

## 1. 背景与目标

当前项目主链路已经具备基本可用性，但仍存在三类问题：

1. 页面功能未闭环，前端有入口或页面草稿，但实际能力不完整
2. 前后端接口未对齐，前端已定义的接口或字段在后端不存在
3. 页面职责重复或定位不清，例如独立统计页与看板页能力重叠

本轮文档目标不是继续罗列问题，而是把缺失功能整理成一份前后端都可直接执行的开发说明，明确：

- 先做什么，后做什么
- 哪些属于前端任务，哪些属于后端任务
- 哪些地方需要双方先对齐字段和行为
- 每个阶段完成后如何联调和验收

本轮已确认的重要决策：

- 删除独立统计页 `StatisticsView`
- 删除 `/statistics` 路由及页面入口
- 原统计能力收敛到 `DashboardView`
- Dashboard 采用中等统计能力，不扩展为复杂分析中心
- 产品主线选择“记录效率优先”，而不是“预算优先”或“账户体系优先”
- 当前不进入账户体系开发，避免账单、预算、转账语义整体重做
- 邮件提醒纳入近期范围，首批包含“催记账邮件”和“月度摘要邮件”
- 设置域需要补齐通知设置、个人资料、数据管理等基础闭环

---

## 2. 优先级划分原则

本次优先级采用 `P0 / P1 / P2` 三层，不按页面热度划分，而按“是否影响核心闭环”和“是否阻塞联调”划分。

### 2.1 P0

P0 处理直接影响主流程可用性的缺口。完成后，用户登录后能够稳定完成快速记账、整理账单、查看轻量复盘、维护分类/预算/资料/通知偏好等核心操作。

P0 的特点：

- 以补齐现有残缺闭环为主
- 优先使用现有能力，少引入新子系统
- 对联调价值最高

### 2.2 P1

P1 处理重要但不阻塞主链路的能力。完成后，产品会从“能用”提升为“更完整、更顺手”，并开始具备通过邮件回流用户的能力。

P1 的特点：

- 明显改善体验，但不是当前上线阻塞项
- 通常需要新增一段较完整的流程

### 2.3 P2

P2 处理增强型能力或范围尚需进一步确认的子系统。完成后，产品会有更强扩展性，但不会作为本轮联调优先目标。

P2 的特点：

- 更接近新增能力域，而不是补缝
- 依赖更多产品决策和更长排期

---

## 3. 优先级总览

| 阶段 | 目标 | 涉及页面 | 前端重点 | 后端重点 | 是否阻塞联调 |
|------|------|----------|----------|----------|--------------|
| P0 | 补齐核心闭环 | `DashboardView` `SettingsView` `Profile` `CategoriesView` `Bills` | 统计页下线、账单整理增强、个人资料页、通知设置页、分类删除对接 | 补分类删除接口、确认统计与用户信息字段、确认通知设置与邮箱字段 | 是 |
| P1 | 补齐留存与回流 | `LoginView` `Bills` `Settings` | 忘记密码闭环、催记账邮件、月度摘要邮件、导出入口 | 补验证码/重置密码接口、邮件任务能力、账单查询增强 | 否 |
| P2 | 规划增强能力 | `manageView` 未来增强页 | 批量整理、默认记账偏好、轻量复盘增强 | 批量处理接口、导出能力增强 | 否 |

## 4. P0 详细设计

### 4.1 阶段目标

删除独立统计页，把真正需要保留的统计能力收敛到 Dashboard，并补齐账单整理、个人资料、通知设置、分类删除几个明显闭环缺口。

### 4.2 页面范围

- `src/views/DashboardView.vue`
- `src/views/StatisticsView.vue`（下线）
- `src/router/index.ts`
- `src/views/SettingsView.vue`
- 新增 `src/views/settings/ProfileView.vue`
- 新增 `src/views/settings/NotificationSettingsView.vue`
- `src/views/settings/CategoriesView.vue`
- `src/views/BillView.vue`
- 相关服务层：`src/services/bill.ts`、`src/services/user.ts`

### 4.3 前端开发项

#### 4.3.1 Dashboard 统计增强

- 保留并强化当前 Dashboard 中的统计模块：
  - 收支概览
  - 分类占比
  - 消费趋势
  - 预算执行
- 增加时间维度切换：
  - 本月
  - 近 3 月
  - 本年
- Dashboard 保持“中等统计”定位，不新增复杂分析中心能力
- Dashboard 中的统计模块应支持：
  - 加载态
  - 空态
  - 接口失败提示
- Dashboard 提供明确的处理入口：
  - 去账单页查看明细
  - 去预算页调整预算
  - 去分类页维护分类

#### 4.3.2 统计页下线

- 删除 `/statistics` 路由
- 删除 Dashboard 或其他页面中所有跳往 `/statistics` 的入口
- 删除或停用 `src/views/StatisticsView.vue`
- 确保底部导航、页面跳转、空链接中不再出现统计页入口

#### 4.3.3 账单整理增强

- `BillView` 从“基础列表页”提升为“账单整理中心”
- 在现有类型、时间筛选基础上补齐：
  - 分类筛选
  - 关键词搜索（至少支持备注、分类名）
  - 真正可用的分页/加载更多交互
- 明确整理结果反馈：
  - 搜索无结果空态
  - 无更多数据结束态
  - 删除/编辑成功后的列表刷新策略
- 保持个人账单整理定位，不引入账户筛选和对账语义

#### 4.3.4 个人资料闭环

- 在设置体系下新增个人资料页面
- 为 `/settings/profile` 新增路由
- 页面应支持：
  - 查看当前用户资料
  - 修改用户名
  - 修改头像 URL
  - 修改个人描述
- 保存后重新进入页面能看到最新数据
- 页面遵循现有设置页风格与错误提示模式

#### 4.3.5 通知设置闭环

- 在设置体系下新增通知设置页面
- 为 `/settings/notifications` 新增路由
- 页面至少支持：
  - 邮箱地址查看与修改
  - 是否开启催记账邮件
  - 是否开启月度摘要邮件
  - 邮件发送时间偏好
- 设置页总入口只承担导航职责，不在卡片页内直接堆复杂通知表单

#### 4.3.6 分类删除对接

- 保持 `CategoriesView` 当前删除交互
- 对接后端真实删除接口
- 根据后端返回的业务失败原因展示明确提示

### 4.4 后端开发项

#### 4.4.1 补齐分类删除接口

- 新增分类删除接口，建议沿用现有命名风格，统一为 `/app/delCategory` 或与前端约定后的唯一命名
- 请求体至少包含分类 `id`
- 删除前需明确校验规则：
  - 若分类已被账单引用，是否允许删除
  - 若分类已被预算引用，是否允许删除
- 本文档建议：
  - 若已被账单或预算引用，则不允许删除
  - 返回明确业务提示，由前端直接展示

#### 4.4.2 确认 Dashboard 统计接口字段

- 确认以下接口返回结构与字段语义，前后端统一后再联调：
  - `/app/getCashFlowCard`
  - `/app/consumerTrends`
  - `/app/consumptionStatistics`
  - `/app/getBudgetInfo`
- 需要重点确认：
  - 时间范围参数在本月、近 3 月、本年下的使用方式
  - 趋势数据字段命名
  - 分类占比中分类名、图标、金额字段
  - 预算执行中的已花费字段命名，是否为 `spent` 或 `cost`

#### 4.4.3 用户信息更新接口确认

- 继续使用 `/user/getUserInfo`
- 继续使用 `/user/updateInfo`
- 后端需确认页面所需字段完整可用：
  - `username`
  - `email`
  - `avatarUrl`
  - `description`

#### 4.4.4 通知设置与邮件字段确认

- 后端需确认是否复用 `/user/updateInfo` 承担通知偏好更新；若不复用，则补单独通知设置接口
- 需要明确的字段包括：
  - `email`
  - `emailReminderEnabled`
  - `monthlySummaryEnabled`
  - `reminderSendHour`
- 若邮件发送由后端任务执行，还需确认：
  - 调度频率
  - 重复发送去重规则
  - 失败重试策略

### 4.5 接口与字段变更

| 能力 | 当前状态 | 处理方式 |
|------|----------|----------|
| 分类删除 | 前端有 `deleteCategory`，后端无对应 controller | 后端补接口，前端统一接口命名 |
| 统计能力 | 后端已有接口，前端独立统计页未接且将下线 | 改为 Dashboard 统一承接 |
| 个人资料更新 | 后端已有 `updateInfo`，前端无页面 | 前端补页面并接入 |
| 通知设置 | 前端无页面，后端字段未确认 | 前后端先统一通知字段和保存方式 |
| 账单整理增强 | 前端已有分页状态但能力未落地 | 前端补界面交互，后端确认搜索/筛选字段 |

### 4.6 联调顺序

1. 后端先确认统计接口字段、分类删除接口命名、通知设置字段
2. 前端开始改造 Dashboard，并移除统计页入口
3. 前端补 `BillView` 整理能力、个人资料页、通知设置页和路由
4. 前后端联调分类删除、用户信息更新、通知设置保存、Dashboard 时间切换统计
5. 最后做回归验证，确保路由、入口、设置保存都完整

### 4.7 验收标准

- 独立统计页已删除，系统内无 `/statistics` 入口
- Dashboard 可按本月、近 3 月、本年查看统计数据
- Dashboard 在空数据场景下显示空态，不报错
- Dashboard 接口失败时有统一错误提示
- 账单页可按分类和关键词整理账单
- 账单页加载更多真正可用，且无更多数据时状态明确
- 设置页中的个人资料入口可正常进入页面
- 设置页中的通知设置入口可正常进入页面
- 修改个人资料后刷新或重新进入页面可看到最新信息
- 修改通知设置后刷新或重新进入页面可看到最新配置
- 分类删除成功时列表刷新
- 分类删除失败时显示明确失败原因

## 5. P1 详细设计

### 5.1 阶段目标

补齐重要体验和回流能力，主要解决登录辅助流程、邮件触达、账单导出等问题。

### 5.2 页面范围

- `src/views/LoginView.vue`
- `src/views/BillView.vue`
- `src/views/settings/NotificationSettingsView.vue`

### 5.3 前端开发项

#### 5.3.1 忘记密码闭环

- 替换当前“开发中”提示
- 忘记密码页保留现有表单结构
- 接入真实发送验证码能力
- 接入真实重置密码能力
- 成功后引导回登录
- 失败时展示明确错误提示

#### 5.3.2 账单页筛选增强

- 在现有类型、时间筛选基础上补充分类筛选
- 本轮不做账户筛选
- 若后续决定纳入账户体系，再作为单独增强项处理

#### 5.3.3 账单分页/加载更多落地

- 将当前已有的 `pageNum`、`pageSize`、`hasMore`、`loadMore` 逻辑真正落到界面交互
- 支持点击加载更多或滚动加载，二选一即可
- 加载更多过程中需显示加载状态
- 无更多数据时需展示明确结束状态

#### 5.3.4 催记账邮件

- 基于通知设置中的邮箱与提醒开关发送邮件
- 默认只做固定频率催记账，不做复杂规则引擎
- 邮件内容包含明确回流入口：
  - 记一笔入口优先回到 `Home`
  - 查看当日/当月账单入口可回到 `Bills`

#### 5.3.5 月度摘要邮件

- 每月固定时间发送个人摘要邮件
- 内容至少包含：
  - 本月收入
  - 本月支出
  - 本月结余
  - 预算执行摘要
  - 主要支出分类
- 邮件中的详情入口统一回到 `Dashboard`

#### 5.3.6 数据导出入口

- 在设置域提供账单导出入口
- 第一阶段只要求账单导出，不扩展到账户、目标等未落地数据域
- 导出失败时提供明确提示，不做静默失败

### 5.4 后端开发项

#### 5.4.1 忘记密码相关接口

- 新增发送验证码接口
- 新增重置密码接口
- 明确验证码有效期、重试限制、错误提示

#### 5.4.2 账单筛选能力确认

- 确认账单列表接口支持按 `categoryId` 查询
- 确认是否支持关键词搜索；若当前接口不支持，则补充按备注/分类名模糊查询能力
- 若前端仅做分类筛选，则当前后端能力已基本满足

#### 5.4.3 邮件任务能力

- 补齐催记账邮件任务与月度摘要邮件任务
- 明确邮件模板字段来源与计算口径
- 明确任务跳过规则，例如：
  - 当天已记账是否跳过催记账邮件
  - 用户未开启对应开关是否跳过

#### 5.4.4 导出能力

- 若前端下载文件走后端生成，则需补账单导出接口
- 需明确导出格式、时间范围、文件命名规则

### 5.5 联调顺序

1. 后端补忘记密码相关接口并确认返回约定
2. 前后端确认邮件任务字段、发送规则、导出方式
3. 前端接入忘记密码流程
4. 前端完善通知设置、账单页筛选与分页交互、数据导出入口
5. 联调忘记密码、邮件提醒、月度摘要、账单列表筛选/分页、导出行为

### 5.6 验收标准

- 忘记密码流程不再是前端模拟
- 用户可完成发送验证码和重置密码操作
- 账单页可按分类筛选
- 账单页支持关键词搜索，且无结果时状态明确
- 账单页加载更多逻辑真正可用
- 无更多数据时页面表现明确且不重复请求
- 用户可在通知设置中开启/关闭催记账邮件和月度摘要邮件
- 催记账邮件命中后可跳回记账或账单整理入口
- 月度摘要邮件内容与 Dashboard 核心数据口径一致
- 用户可完成账单导出，失败时看到明确提示

## 8. 风险与待决策项

### 8.1 分类删除规则

当前最大不确定点是：分类被账单或预算引用时，是否允许删除。

本文档建议：

- 不允许级联删除历史数据
- 如分类被引用，则直接阻止删除，并返回明确提示

### 8.2 账户体系是否进入近期开发

账户体系会牵动：

- 账单实体
- 账单查询条件
- 管理页面入口
- 账户树与层级关系

该项已在本轮明确为“不进入近期开发”，避免与账单、预算、提醒能力混排。

### 8.3 忘记密码与邮件提醒能力边界

如果忘记密码使用邮件验证码，则需确认：

- 邮件能力是否稳定可用
- 发送频率限制
- 验证码过期时间
- 失败场景提示文案

同时需要区分两类邮件：

- 忘记密码验证码邮件：安全链路，强调时效和限制
- 产品提醒邮件：业务链路，强调回流和去重

### 8.4 Dashboard 范围控制

独立统计页删除后，需要明确 Dashboard 只承接中等统计能力，不继续扩展为复杂报表中心。复杂对比分析、年度深度报表、多维交叉筛选均不进入本轮。

### 8.5 邮件提醒打扰控制

邮件提醒必须避免打扰过度，至少需要明确：

- 催记账邮件的默认频率
- 同一天是否重复发送
- 已记账用户是否跳过提醒
- 用户关闭提醒后的立即生效规则

## 9. 开发顺序建议

建议按以下顺序推进：

1. 后端先补齐 P0 中的接口与字段确认
2. 前端同步开始 Dashboard 改造、统计页下线、Bills 整理能力增强
3. 前端补个人资料页与通知设置页，后端配合用户信息和通知字段联调
4. 完成 P0 联调与回归后，再进入 P1
5. P1 完成忘记密码、邮件提醒、月度摘要、数据导出
6. P2 仅保留轻量增强，不引入账户体系立项

## 10. 错误处理与验收补充

### 10.1 错误处理原则

- 沿用当前统一消息提示模型，不新增另一套错误处理体系
- 解析记账失败时，仍需允许用户直接进入手动补录表单，不能中断主链路
- 账单搜索、筛选、加载更多失败时，需保留当前筛选条件，便于重试
- 通知设置保存失败时，页面需保留用户已编辑内容，避免全部重填
- 邮件任务失败不应影响前台页面可用性，但应能在后端日志或任务记录中定位

### 10.2 主链路验收口径

- 用户可以从 `Home` 快速记一笔，并在保存后继续记下一笔
- 用户可以在 `Bills` 完成查找、筛选、编辑、删除、继续补录
- 用户可以在 `Dashboard` 完成月度轻量复盘，而无需进入独立统计页
- 用户可以在 `Settings` 完成资料维护、通知偏好维护和基础数据管理入口访问
- 用户开启催记账邮件后，提醒邮件可把用户回流到记账或账单整理入口
- 用户开启月度摘要邮件后，摘要内容与 Dashboard 关键指标口径一致

### 10.3 不纳入本轮验收范围

- 账户管理
- 转账语义建模
- 独立提醒中心
- 独立统计/报表页
- 多维高级分析

## 11. 可执行写法标准

为避免这份文档停留在概念层，后续执行时统一采用以下写法标准：

- 前端任务写到页面和动作级，不写“优化页面”这种泛描述
- 后端任务写到接口和字段级，不写“补接口”这种泛描述
- 联调项明确谁先完成、谁后接入、哪些字段先对齐
- 验收标准写到可观察结果级，不写“功能正常”这类抽象描述
- 风险项单独列出，不与已确认开发项混写

按本标准拆解后，前后端可以直接依据文档开发，不需要再做一次范围重构。
